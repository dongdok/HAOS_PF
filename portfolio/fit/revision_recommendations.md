# 포트폴리오 보강 권장안 (아카라라이프 지원 기준)

> 원칙: 과장 없이, 확인 가능한 사실만, 담백한 문장으로 작성
> 기준: 사람인 공고(스마트홈 시스템 엔지니어) + 현재 이력서/경력기술서

## 1) 현재 문서에서 상대적으로 약한 부분

## A. "설치/시운전" 실무 흐름이 짧게 보임
- 현재도 런북은 있지만, 현장 엔지니어가 실제로 보는 체크리스트 형태가 더 필요합니다.
- 면접관 입장에서는 "실제 오픈 전에 무엇을 어떤 순서로 확인했는지"를 빠르게 보고 싶어 합니다.

보강 제안
- 시운전 문서에 "필수 10개 체크 항목"을 추가
- 각 항목에 "실패 시 다음 조치"를 1줄로 명시

## B. "장애 대응"의 전/후 비교가 부족함
- RCA는 잘 되어 있지만, 조치 전후 차이를 숫자/로그 예시로 보여주면 설득력이 더 커집니다.

보강 제안
- 장애 사례마다 아래 3개를 통일 표기로 추가
  - 증상 로그 1줄
  - 조치 내용 1줄
  - 조치 후 확인 결과 1줄

## C. "협업 방식"이 결과 중심으로만 보임
- 현재 문서는 기술 내용은 좋지만, 협업 커뮤니케이션 흐름(누가/언제/무엇을)도 드러나면 실무 적합성이 높아집니다.

보강 제안
- "이슈 티켓 템플릿" 1개 추가
  - 현상
  - 재현 조건
  - 요청사항
  - 완료 기준

## D. "업무 범위 경계"가 더 명확하면 좋음
- 본인이 직접 수행한 것과 협력사/타 플랫폼 담당 영역을 분리하면 신뢰도가 더 높아집니다.

보강 제안
- 프로젝트 요약 문서에 "내가 직접 한 일 / 협업으로 해결한 일" 2개 섹션 분리

## 2) IoT 네트워크 엔지니어 관점에서 추가하면 좋은 요소

## A. L2/L3 기본 정보 1장
- 내부망 대역, 게이트웨이, DHCP 예약 정책, VPN 접근 경로를 한 페이지로 요약
- 복잡한 설정보다 "운영 원칙"을 짧게 쓰는 게 효과적입니다.

## B. 무선 간섭 대응 기준 1장
- Zigbee 채널 변경 이유
- Wi-Fi 2.4GHz와의 겹침 회피 원칙
- 변경 후 검증 방법(재조인, 라우트 안정화 확인)

## C. 엔티티 재등록 표준 절차 1장
- 재등록 전 백업 항목
- 재등록 후 복구 순서(디바이스명 → 엔티티ID → 자동화/스크립트 → 대시보드)
- 실수하기 쉬운 포인트 3개

## 3) 문장 톤 가이드 (지금처럼 유지 권장)

좋은 표현 예시
- "~를 구축했다"보다 "~를 적용해 운영했다"
- "완벽히 해결"보다 "재현 후 원인 분리해 조치했다"
- "최고 성과"보다 "기준 대비 개선을 확인했다"

피하면 좋은 표현
- "혁신", "압도", "완전 자동화", "무결점"
- 근거 없는 퍼센트 개선치

## 4) 바로 적용 가능한 수정 우선순위

1. 시운전 문서에 체크리스트 강화
2. RCA 문서에 전/후 비교 3줄 포맷 추가
3. 프로젝트 요약에 "직접 수행/협업" 경계 명시
4. 협업 티켓 템플릿 샘플 추가

## 5) 최종 점검 체크리스트

- 모든 수치에 기준일이 있는가
- 로그 예시는 실제 문자열 기반인가
- "내가 한 일"과 "협업한 일"이 섞여 있지 않은가
- 문장에 과장 표현이 없는가
- 면접관이 3분 내 핵심을 읽을 수 있는가
