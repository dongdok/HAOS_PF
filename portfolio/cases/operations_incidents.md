# 운영 장애 사례 (공개용)

## 사례 1) Zigbee 라우팅 오류/타임아웃

- 증상
  - Z2M 로그에 `ROUTE_ERROR`, `Delivery failed`, `ZCL command timeout` 반복
  - 특정 조명/스위치가 간헐적으로 제어 실패

- 원인
  - 2.4GHz 대역 간섭 + 메쉬 경로 불안정
  - 일부 기기 재등록 이후 라우팅/참조 정합성 저하

- 조치
  - Zigbee 채널 재조정
  - 문제 기기 재조인
  - 디바이스명/엔티티명/자동화 참조 동기화

- 결과
  - 명령 실패 빈도 감소
  - 핵심 자동화 경로 정상 복구
  - 검증: Zigbee2MQTT 로그에서 timeout/route error 빈도 추이 확인, 자동화 실행 이력 정상화 확인

## 사례 2) 재등록 후 자동화 미동작

- 증상
  - 기기는 정상 조인되었으나 자동화가 해당 기기를 못 찾음
  - 대시보드에 “엔티티를 찾을 수 없음” 경고 발생

- 원인
  - 재등록으로 엔티티 ID/디바이스 식별자 변경
  - 기존 스크립트/자동화가 구 엔티티 참조 유지

- 조치
  - 캐노니컬 명칭으로 디바이스명/엔티티명 복구
  - 관련 자동화/스크립트의 참조 일괄 업데이트
  - 대시보드 카드 참조 동기화

- 결과
  - UI/자동화/스크립트 모두 정상 연동
  - 재등록 표준 절차 문서화 완료
  - 검증: 대시보드 경고 해소, 자동화 트레이스에서 대상 엔티티 정상 참조 확인

## 사례 3) 중복 명령 전송으로 인한 지연

- 증상
  - 동일 대상 ON/OFF 명령이 중복 전송되어 로그 노이즈 증가
  - 일부 스위치/조명에서 타임아웃 발생 확률 증가

- 원인
  - 상태 확인 없이 제어 명령을 반복 전송하는 로직

- 조치
  - 자동화에 “현재 상태 확인 후 명령 전송” 조건 추가
  - 이미 목표 상태면 명령 스킵

- 결과
  - 메시지 부하 감소
  - 불필요한 재시도/실패 로그 감소
  - 검증: 동일 구간 대비 중복 ON/OFF 명령 수 및 delivery fail 로그 감소 여부 확인

## 운영 원칙

- 실패를 숨기지 않고 원인 분류 후 복구
- 임시 우회보다 참조 정합성 복구를 우선
- 복구 후 재발 방지 조건까지 반영
