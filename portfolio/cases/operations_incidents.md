# 운영 장애 및 조치 이력

## 1) 지그비 통신 지연 및 제어 실패

- **증상**: 특정 조명이나 스위치가 가끔씩 제어되지 않거나, 실행 로그에 통신 오류가 반복해서 기록됨.
- **원인**: 집안의 2.4GHz 와이파이 대역과의 간섭, 그리고 일부 기기 재등록 후에 통신 경로 설정이 불안정해진 것이 원인.
- **조치**:
  - 지그비 채널을 와이파이 간섭이 적은 대역으로 재설정.
  - 연결이 불안정한 기기들을 네트워크에 다시 조인(Join) 시킴.
  - 기기 이름을 표준 규칙에 맞춰 다시 정리하고 자동화 연결 상태를 복구.
- **결과**: 제어 실패 빈도가 줄어들고, 핵심 자동화가 정상적으로 작동함.
- **검증**: 로그상에서 통신 오류 빈도가 낮아진 것을 확인하고, 실제 조명 동작 상태를 최종 점검함.

## 2) 기기 재등록 후 자동화 중단

- **증상**: 기기는 정상적으로 연결되었으나 기존에 만들어둔 자동화가 해당 기기를 인식하지 못함. 대시보드 이미지에도 "엔티티를 찾을 수 없음" 경고가 발생.
- **원인**: 기기를 다시 등록하면서 내부 식별값(Entity ID)이 바뀌었는데, 기존 자동화나 스크립트가 옛날 이름을 참조하고 있었기 때문.
- **조치**:
  - 새로 등록된 기기의 이름을 기존에 정해둔 표준 이름으로 다시 변경.
  - 자동화와 스크립트에서 이름이 깨진 참조 부분들을 일괄 수정.
  - 대시보드 화면의 연결 정보도 실시간으로 동기화.
- **결과**: 화면과 자동화 모두 정상화됨. 이후 '재등록 시 이름 맞추기'를 표준 절차로 정립함.
- **검증**: 대시보드 경고 해제 여부와 자동화 실행 기록(Trace)의 정상 참조 여부를 확인.

## 3) 중복 명령 전송으로 인한 네트워크 부하

- **증상**: 동일한 켜짐/꺼짐 명령이 짧은 시간에 여러 번 전송되어 시스템 로그가 지저분해지고, 간혹 명령 처리가 늦어지는 현상 발생.
- **원인**: 기기의 현재 상태를 확인하지 않고 무조건 명령만 보내도록 설계된 자동화 로직 때문.
- **조치**:
  - 모든 자동화에 "현재 상태를 먼저 확인"하는 조건을 추가.
  - 이미 켜져 있는 기기라면 켜기 명령을 중복으로 보내지 않고 종료하도록 수정.
- **결과**: 불필요한 통신 메시지가 크게 줄어들고 전체적인 시스템 반응 속도가 안정됨.
- **검증**: 동일 조건에서 중복 전송되는 로그가 사라진 것을 확인.

## 운영 원칙

- 장애를 숨기지 않고 원인을 정확히 분류해서 기록함.
- 임시방편보다 기기 이름(참조)을 원래대로 복구하는 근본적인 방법을 우선함.
- 복구 후에는 비슷한 문제가 반복되지 않도록 자동화 조건에 방어 로직을 반영함.
